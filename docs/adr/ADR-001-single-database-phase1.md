# ADR-001: Phase 1 采用单库策略

## 状态
已接受

## 日期
2026-06-23

## 背景

MIS Platform 规划为微服务架构，但 Phase 1 需要快速交付认证、用户、组织、权限等基础能力。若 Phase 1 即为每个微服务分配独立数据库，将引入：

- 跨服务分布式事务（Seata）复杂度
- 联调与环境搭建成本上升
- Flyway 迁移脚本分散管理

## 决策

**Phase 1 所有 Java 微服务共享单一 PostgreSQL 数据库 `mis_platform`**，按表前缀（`sys_`）和模块归属划分逻辑边界。服务间通过 WebClient/RestClient 调用而非跨库 JOIN（见 ADR-007）。

Phase 2 视业务增长与团队成熟度，再按限界上下文拆库。

## 备选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| A. 单库（选定） | 简单、无分布式事务 | 服务边界可能被侵蚀 |
| B. 每服务一库 | 边界清晰 | Phase 1 过重 |
| C. 模块化单体 | 最简单 | 与微服务规划不一致 |

## 后果

### 正面
- 开发速度快，本地环境简单
- 避免 Phase 1 引入 Seata
- 联调问题少

### 负面
- 服务可能 temptation 直接访问他模块表（需 Code Review 约束）
- 未来拆库需要数据迁移

## 约束

- 每个服务只访问本模块负责的表
- 跨模块数据通过内部 HTTP API 获取（RestClient / WebClient）
- 实体类不跨模块共享（DTO/VO 在 API 层转换）

- [ ] 是否在 pom 中用 ArchUnit 强制包依赖规则
